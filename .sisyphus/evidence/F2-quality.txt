=== Manual Review Findings ===

ISSUE 1: expect() in production code [CRITICAL — violates criterion #4]
  File: src/log/file.rs:40
  Code: let path = config.log_file.as_ref().expect("file layer requires log_file");
  Details: build_file_layer() is only called when config.log_file.is_some(), so this is
           logically unreachable, but it is still an expect() in library production code.
           Should use an early return or propagate an error instead.

ISSUE 2: Dead code suppressed with #[allow(dead_code)] [violates criterion #3]
  a) src/log/console.rs:14   — min_level field is never read (render_event uses current_min_level() from init.rs)
  b) src/log/console.rs:17  — tz_warning_emitted field is never read
  c) src/log/console.rs:22  — ConsoleLayer::new() is never called (init.rs uses new_with_tz())
  d) src/log/console.rs:110 — resolve_tz() is never called (init.rs has its own resolve_tz())
  e) src/log/file.rs:27     — min_level field is never read
  Details: These are genuinely unused. The correct fix is to remove them, not silence the compiler.
           The duplicate resolve_tz() in console.rs should be deleted entirely.

ISSUE 3: Inconsistent match vs let-else style [violates criterion #1]
  File: src/log/file.rs:92
  Code: let event_level = match Level::from_tracing(*meta.level()) {
            Some(l) => l,
            None => return Vec::new(),
        };
  Details: src/log/console.rs:51 uses the more idiomatic let-else pattern for the exact same
           operation: let Some(level) = Level::from_tracing(*tracing_level) else { return Vec::new(); };
           file.rs should match console.rs for consistency.

ISSUE 4: Minor allocation inefficiency [style, non-blocking]
  File: src/log/console.rs:62
  Code: msg.push_str(&format!(" {}={}", k, v));
  Details: Creates a temporary String per field inside a loop. Could use write!(&mut msg, ...)
           to append directly. Not a clippy warning, but worth cleaning up.
